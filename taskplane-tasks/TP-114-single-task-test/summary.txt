Taskplane Runtime V2 is a proposed redesign of the control plane that powers Taskplane's orchestrated task execution. While the durable data plane — including STATUS.md, .DONE markers, batch-state.json, and the dependency DAG/waves/lanes/worktrees model — is preserved, Runtime V2 replaces the fragile runtime infrastructure built on TMUX, sidecar file tailing, and terminal-session coupling. The core diagnosis is that the current system has too many overlapping layers of indirection, creating ambiguous ownership, transport inversion, and a terminal dependency that makes task outcomes unreliable.

The new architecture introduces three deterministic Node process components: the engine process (which handles discovery, DAG planning, wave orchestration, and batch state), the lane-runner process (which owns serial task execution within a lane, STATUS parsing, step progression, and .DONE handling), and the agent-host process (which adapts the Pi subprocess lifecycle, normalizes telemetry, handles mailbox/steer injection, and persists structured event logs). Each spawned process has exactly one deterministic owner, eliminating the ambiguity of the current multi-layered nesting. The dashboard server is updated to render agent conversations and telemetry from these structured logs rather than relying on TMUX pane capture.

Runtime V2 is architecturally complete when worker, reviewer, and merge lifecycles no longer depend on TMUX; when the dashboard can show agent conversations and telemetry without pane capture; when send_agent_message and mailbox features operate on process-registry-backed agent IDs; and when the lane execution path no longer depends on the task-runner.ts session lifecycle. The /task command becomes a compatibility wrapper or is removed in a future major version, with /orch serving as the authoritative execution path even for single tasks. Key invariants — file-backed memory, resumable batch state, and mailbox-based steering — are preserved throughout.
