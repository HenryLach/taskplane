Runtime V2 is an architectural redesign of Taskplane's control plane that replaces the fragile TMUX-based session management with a deterministic Node.js process hierarchy. The core motivation is that the original runtime suffered from deeply nested process layers (up to 10 levels deep), ambiguous process ownership, transport inversion where sibling processes communicated through file polling instead of direct IPC, and a correctness dependency on terminal infrastructure like TMUX. Runtime V2 keeps Taskplane's proven durable data plane — PROMPT.md, STATUS.md, .DONE files, batch-state.json, the dependency DAG, waves, lanes, and worktrees — while completely replacing how processes are spawned, owned, and monitored.

The new architecture introduces a clean four-layer hierarchy: the extension (command surface) owns the engine process, the engine owns lane-runners and merge hosts, each lane-runner owns its worker and reviewer agent-hosts, and each agent-host owns its Pi subprocess. The key invariant is that every spawned process has exactly one deterministic owner. Lane-runners are a new component that replace the previous pattern of running Pi sessions with the task-runner extension; they handle the task execution state machine, STATUS parsing, step progression, checkpoint discipline, and .DONE handling for a single lane worktree. Agent-hosts are a new transport adapter around `pi --mode rpc` that handle subprocess lifecycle, live RPC event parsing, telemetry normalization, mailbox steering injection, and conversation persistence.

Runtime V2 also introduces a batch-scoped storage layout under `.pi/runtime/{batchId}/` for agent manifests, event streams, and lane snapshots. Steering shifts entirely to a mailbox-based model where the supervisor writes to agent inboxes and agent-hosts inject messages via Pi RPC, eliminating any need for TMUX pane capture. The dashboard reads normalized structured event logs rather than relying on terminal output. The `/task` command becomes a compatibility wrapper around the same headless execution core, with `/orch` as the authoritative execution path even for single tasks. Success is defined as the complete removal of TMUX dependencies from worker/reviewer/merge lifecycle, full dashboard observability without pane capture, and explicit packet-path authority throughout the system.
